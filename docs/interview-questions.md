# MiniRPC 面试题与参考答案

这份文档不是“关键词提纲”，而是尽量按真实面试口径整理的可直接回答版本。回答内容都围绕当前项目实现展开，便于你在面试中把“做了什么、为什么这么做、踩过什么坑、如何验证效果”讲完整。

---

## 一、项目整体认知

### 1. 先整体介绍一下你的 RPC 项目。

参考回答：

我做的是一个简化版的 C++ RPC 框架，名字叫 `MiniRPC`，目标不是替代生产级 gRPC 或 bRPC，而是完整实现一条 RPC 调用链路，并把里面最关键的工程点做出来。

这个项目主要包含 6 个部分：

1. **IDL 与代码生成**
   - 我定义了一套 protobuf 风格的精简 IDL，文件在 `idl/calculator.mproto`
   - 用 Python 写了生成器 `tools/mrpcgen.py`
   - 它会自动生成消息结构体、序列化/反序列化函数、客户端 Stub、服务端注册胶水代码

2. **自定义协议**
   - 我定义了固定长度协议头 `FrameHeader`
   - 包头里有魔数、版本、消息类型、请求 ID、body 长度
   - 包体里有 service 名、method 名、auth token、deadline 和业务 payload

3. **网络通信**
   - 底层用 TCP
   - 客户端是 `RpcChannel`
   - 服务端是 `RpcServer`
   - 支持连接复用、超时设置、`TCP_NODELAY`

4. **服务注册与发现**
   - 当前版本是静态注册发现
   - 客户端从 `configs/registry.ini` 查服务地址
   - 服务端配置在 `configs/server.ini`

5. **并发处理和保护**
   - 服务端是 accept + 连接处理 + 业务线程池的结构
   - 线程池队列是有界的，满了直接返回 `BUSY`

6. **测试和基准**
   - 我写了运行时单测和端到端集成测试
   - 还写了 benchmark，统计 QPS、平均延迟、P50/P95/P99，并输出到 `docs/perf-report.md`

这个项目最大的价值是：我不是只写了一个 socket demo，而是把 RPC 的协议、Stub、序列化、调度、错误处理、性能测试这些核心能力都串起来了。

---

### 2. 你为什么要做一个自己的 RPC 框架，而不是直接用 gRPC？

参考回答：

我做这个项目的目的主要是**学习和展示底层理解**，不是重复造轮子去替代成熟框架。

如果直接用 gRPC，我当然可以很快搭一个服务出来，但那样更多体现的是“会使用框架”。而这个项目更想体现的是：

- 我理解 RPC 的调用本质：本地方法调用如何映射到网络调用
- 我知道协议头为什么要设计成固定长度
- 我知道 TCP 字节流怎么处理粘包拆包
- 我知道为什么需要 IDL 和代码生成
- 我知道并发、过载、超时、鉴权这些能力在框架层怎么落地

所以这个项目更像是“自己实现一个可运行、可测试、可解释的 RPC 最小闭环”。这样在面试里，我能讲的不只是“我用过什么”，而是“我实现过什么、为什么这么实现、出现过什么问题、怎么优化的”。

---

### 3. 你这个项目的边界是什么？哪些能力做了，哪些故意没做？

参考回答：

我一开始就把项目边界定义得比较清楚：**做完整闭环，不追求全量生产能力**。

已经做的能力包括：

- IDL 定义
- 自动生成客户端和服务端代码
- 自定义二进制协议
- TCP 同步调用
- 静态服务发现
- 鉴权 token
- IP 白名单
- deadline / timeout
- 线程池并发执行
- 有界队列快速拒绝
- 测试和 benchmark

没有做或者只留了扩展点的包括：

- 真正的 Protobuf 编译器插件
- 异步 RPC / future / callback
- 流式 RPC
- epoll/reactor
- etcd / ZooKeeper / Consul
- 重试、熔断、限流、追踪、指标系统

我故意这么做，是为了在复杂度可控的前提下，把最能体现底层理解的部分做扎实。

## 二、IDL 与代码生成

### 4. 为什么 RPC 框架通常都要有 IDL？

参考回答：

IDL 的核心作用是把“接口定义”和“实现细节”分离。

如果没有 IDL，客户端和服务端就得手写很多重复代码，比如：

- 请求和响应结构
- 序列化和反序列化
- 网络请求拼包
- 服务名和方法名绑定

这样的问题是：

1. 容易出错
2. 客户端和服务端约定难以统一
3. 接口演进没有中心定义

我这个项目里，IDL 文件 `idl/calculator.mproto` 就是接口契约。生成器 `tools/mrpcgen.py` 会根据它生成：

- 消息结构体
- `SerializeToString` / `DeserializeFromString`
- 服务抽象类
- 客户端 Stub
- 服务端注册函数

所以业务开发只需要：

- 服务端继承生成的抽象基类，实现业务方法
- 客户端直接调用生成的 Stub

这就是典型的“契约驱动开发”。

---

### 5. 你的代码生成器生成了哪些代码？分别解决什么问题？

参考回答：

生成器主要生成四类代码：

1. **消息结构体**
   - 比如 `AddRequest`、`AddResponse`
   - 这部分提供强类型数据结构，避免手写 map 或裸二进制解析

2. **序列化 / 反序列化函数**
   - `SerializeToString`
   - `DeserializeFromString`
   - 解决对象到二进制之间的转换问题

3. **客户端 Stub**
   - 比如 `CalculatorServiceClient::Add`
   - 业务方看起来像是在调用本地方法，但底层会自动序列化请求、发网络调用、反序列化响应

4. **服务端注册胶水代码**
   - 比如 `RegisterCalculatorService`
   - 它会把 `service + method` 绑定到具体的 handler
   - 同时负责把网络请求 payload 解成 request 对象，再把 response 对象序列化回去

这样做的好处是，框架层的重复逻辑只写一次，业务层关注实现即可。

---

### 6. 为什么你没有直接使用真正的 Protobuf？

参考回答：

主要是出于项目目标和复杂度控制的考虑。

如果直接接入真正的 Protobuf，我当然能得到更成熟的 IDL、字段兼容性和跨语言能力，但这个项目的重点是：

- 展示我对“代码生成流程”本身的理解
- 展示我对消息编解码、Stub、服务注册这些机制的掌握

如果一开始就把编译器插件、Protobuf runtime、外部依赖都接进来，项目的底层实现感会被冲淡，而且开发复杂度会大幅上升。

所以我当前是做了一个 **protobuf 风格的精简子集**，先把核心机制跑通。后续完全可以演进为：

- 保留当前运行时框架
- 把 `.mproto` 替换成标准 `.proto`
- 再实现 `protoc` 插件或对接现有生成链路

这个演进路径在面试里也很好讲：先从原理实现出发，再往工程化升级。

---

### 7. 你的 IDL 解析器支持哪些能力？限制是什么？

参考回答：

当前支持的是一个精简子集，主要包括：

- `package`
- `message`
- `service`
- `rpc ... returns (...)`

支持的字段类型有：

- `bool`
- `int32`
- `int64`
- `uint32`
- `uint64`
- `double`
- `string`

目前的限制主要有：

- 不支持 `repeated`
- 不支持 `map`
- 不支持嵌套 message
- 不支持 optional/default value 语义
- 不支持字段 tag 的兼容性演进策略

这部分是有意简化的，因为项目目的是体现机制，不是重写完整的 protobuf 编译器。

## 三、协议设计与序列化

### 8. 你的协议头为什么要设计成固定长度？

参考回答：

固定长度头部的核心价值是：**在 TCP 字节流里快速、稳定地确定消息边界**。

我的协议头里包含：

- `magic`
- `version`
- `message_type`
- `codec`
- `flags`
- `request_id`
- `body_length`

固定长度头的好处有几个：

1. **读取逻辑简单**
   - 服务端先读满固定 20 字节头部
   - 再根据 `body_length` 去读 body

2. **便于快速校验**
   - `magic` 可以防止乱流量或协议错连
   - `version` 可以支持未来协议升级

3. **便于性能优化**
   - 协议头解析简单，CPU 开销低

如果头部也是变长的，解析复杂度会更高，而且在网络边界处理上更容易出错。

---

### 9. 你的包体里为什么不只放 payload，还要放 service 和 method？

参考回答：

因为这个项目实现的是一个通用 RPC 框架，而不是某个固定服务的点对点通信。

服务端收到请求之后，必须知道：

- 这是哪个服务：`service`
- 调的是哪个方法：`method`
- 这个请求是谁发来的：`auth_token`
- 这个请求是否已经超时：`deadline_unix_ms`
- 业务参数是什么：`payload`

如果没有 `service + method`，服务端就无法做统一分发。那就退化成了“一个端口只处理一个固定逻辑”，失去了 RPC 框架抽象的意义。

---

### 10. 你的序列化格式为什么用自定义二进制，而不是 JSON？

参考回答：

我做这个选择主要考虑三点：

1. **性能**
   - 二进制编码比 JSON 更紧凑
   - 少了文本解析和字符串转数字的成本

2. **可控性**
   - 这个项目要展示我对序列化机制的理解
   - 自定义二进制格式更适合讲编码顺序、长度前缀、大端序等细节

3. **和 RPC 场景更贴合**
   - RPC 内部传输通常更偏向高效和紧凑，而不是人类可读性

我这里对 `string` 的编码是 `uint32 length + bytes`，数值类型按固定宽度大端序写入。这样实现简单、解析稳定，也符合这个项目“轻量但完整”的定位。

---

### 11. 你怎么处理 TCP 粘包拆包问题？

参考回答：

TCP 本质上是字节流，没有消息边界，所以不能假设一次 `recv` 就对应一个完整请求。

我的处理方式是：

1. 先读取固定长度的 `FrameHeader`
2. 从 header 中解析出 `body_length`
3. 再循环读取指定长度的 body，直到读满

服务端和客户端都实现了 `ReadFully`，保证按字节数读满。这种做法是处理 TCP 粘包拆包最典型也最稳定的方式。

重点是：**判断一条消息是否完整，不能依赖系统调用次数，只能依赖协议长度字段。**

---

### 12. 为什么要加 `magic` 和 `version`？

参考回答：

`magic` 和 `version` 都是典型的协议健壮性设计。

- `magic` 用于快速判断这是不是我定义的协议
  - 比如端口被误连、收到脏数据、或客户端版本不一致时，可以快速失败

- `version` 用于协议演进
  - 当前版本是 `1`
  - 后面如果我增加压缩标记、扩展 header 字段或变更编码方式，就可以通过版本做兼容处理

这两个字段虽然简单，但很像真实中间件里会有的设计，所以对面试很加分。

---

## 四、网络通信与并发模型

### 13. 你的客户端 `RpcChannel` 做了什么事情？

参考回答：

`RpcChannel` 是客户端发起 RPC 的核心组件，主要职责有：

1. 维护到服务端的 TCP 连接
2. 把调用参数封装成请求帧
3. 设置 socket 读写超时
4. 发送请求并等待响应
5. 校验响应帧是否合法
6. 在异常时重置连接，避免脏连接复用

一次调用的大致流程是：

- 先 `EnsureConnected`
- 组装 `RequestFrame`
- 把 header 和 body 编码后发送
- 读取响应 header
- 再读取响应 body
- 解析成 `ResponseFrame`
- 最终返回 `RpcStatus`

从业务角度看像本地函数调用，但底层其实做了完整的网络交互。

---

### 14. 为什么你的客户端要做连接复用？

参考回答：

如果每次 RPC 调用都重新建 TCP 连接，代价会很高，包括：

- 三次握手的时延
- socket 创建和销毁成本
- 内核资源开销

所以我在 `RpcChannel` 里采用的是**长连接复用**：

- 第一次调用时建立连接
- 后续继续复用同一个 socket
- 只有在读写失败、超时或协议异常时，才主动 `Reset`

这样做能显著降低单次调用开销，benchmark 里也能看出小请求场景的收益。

---

### 15. 你的服务端为什么不是单线程？并发模型是什么？

参考回答：

如果服务端是单线程模型，那么：

- 一个慢请求会阻塞后续请求
- 吞吐上不去
- 无法体现 RPC 框架最基础的并发处理能力

我当前服务端模型分三层：

1. **accept 线程**
   - 专门负责接收新连接

2. **连接处理线程**
   - 每建立一个连接，就启动一个线程循环读这个连接上的请求
   - 负责协议解析和响应回写

3. **业务线程池**
   - 真正的业务逻辑不在线程里直接跑
   - 而是提交到固定大小线程池

这样分层的好处是：

- 网络收发和业务执行解耦
- 业务慢不会直接把 accept 卡死
- 线程池可控，便于过载保护

---

### 16. 为什么连接处理和业务执行要分开？

参考回答：

因为它们关注点不一样。

连接处理更偏 IO：

- 读 socket
- 解协议
- 回写响应

业务执行更偏 CPU / 逻辑：

- 真正的服务方法调用
- 可能有计算、阻塞或业务分支

如果把业务逻辑直接跑在连接线程里，会有两个问题：

1. 一个连接上的慢请求会拖慢该连接后续请求处理
2. 连接线程数量会直接和业务并发强绑定，资源不可控

所以我把业务逻辑提交到线程池执行，连接线程只负责收发和桥接，这样职责更清晰。

---

### 17. 你的线程池为什么要有队列上限？

参考回答：

有界队列的核心是做**过载保护**。

如果队列无限增长，当服务处理不过来时，请求就会一直堆积，表现出来是：

- 延迟越来越大
- 内存占用不断上涨
- 用户最终看到的不是“明确失败”，而是“越来越慢”

我在这个项目里选择了一个更合理的策略：

- 线程池任务队列有最大长度
- 当队列满时，`Submit` 直接失败
- 服务端立即返回 `StatusCode::kBusy`

这样调用方可以快速感知系统繁忙，而不是傻等。这是 RPC 框架和服务治理里很重要的一种设计思想。

---

### 18. 你的过载保护在数据上怎么体现？

参考回答：

我专门写了一个过载 benchmark：`rpc_delay_overload_c16`。

这个场景里：

- 服务端线程数只有 2
- 队列上限很小
- 客户端并发是 16
- 服务方法内部会 sleep 20ms，模拟慢请求

实际跑出来的结果是：

- 总请求数 160
- 成功 100
- `BUSY` 60
- 没有无限排队拖死

这说明有界线程池 + 快速拒绝是有效的，也证明项目不是只做功能，还考虑了高并发下的稳定性。

---

## 五、超时、鉴权与异常处理

### 19. 你的超时机制是怎么设计的？

参考回答：

我做了两层超时控制：

1. **客户端 socket 超时**
   - 每次调用前对 socket 设置 `SO_RCVTIMEO` 和 `SO_SNDTIMEO`
   - 如果网络读写卡住，客户端不会无限等待

2. **请求级 deadline**
   - 请求体里带 `deadline_unix_ms`
   - 服务端在执行前会判断 deadline 是否已经过期

这种双层设计的意义是：

- 网络层能处理 IO 卡住
- 框架层能感知请求语义超时

虽然当前版本没有做服务端执行中的抢占取消，但这个设计已经把 timeout 的关键思路体现出来了。

---

### 20. 为什么客户端一旦超时或读写失败就要 `Reset` 连接？

参考回答：

因为在同步长连接模型下，客户端和服务端之间的请求/响应顺序是一一对应的。

如果一个请求已经超时，但服务端稍后又把响应写回来了，这个响应就可能污染下一次调用，导致：

- 响应错位
- request_id 对不上
- 后续解析出错

为了避免连接进入“不可信状态”，我在超时、读写失败、响应帧非法时直接 `Reset`，下次重新建连接。

这是一种比较保守但很安全的做法，在简化版框架里很合理。

---

### 21. 你的鉴权为什么选 token，而不是更复杂的机制？

参考回答：

这个项目里，鉴权的目标是**体现安全意识和框架接入点**，不是做完整安全体系。

所以我选择了最轻量的 token 方案：

- 服务端配置 `auth_token`
- 客户端请求里携带 token
- 服务端分发前校验
- 不一致直接返回 `kAuthFailed`

这样设计的好处是：

- 实现简单
- 能清晰展示“框架层鉴权入口”
- 后续容易演进为 HMAC、签名、双向 TLS 或 OAuth 之类更完整的方案

如果一开始就做太复杂的安全方案，项目重点会偏离 RPC 核心链路。

---

### 22. IP 白名单是怎么做的？它解决什么问题？

参考回答：

服务端在接收连接时会获取对端 IP，然后用配置里的 allowlist 做校验。

如果不在白名单中，就直接关闭连接。

它的作用主要是：

- 提供一个最基础的网络层访问控制
- 体现服务端的安全防护意识

当然它不是强安全能力，因为 IP 可以被代理层隐藏或变更，所以更适合作为一层简单防护，而不是唯一安全手段。

---

### 23. 你是怎么处理服务不存在、方法不存在、请求非法这些异常的？

参考回答：

我统一用了 `RpcStatus` 和 `StatusCode` 表达框架级错误。

比如：

- 找不到服务：`kUnknownService`
- 找不到方法：`kUnknownMethod`
- 反序列化失败：`kBadRequest`
- 序列化失败：`kSerializeError`
- 鉴权失败：`kAuthFailed`
- 线程池满：`kBusy`
- 网络异常：`kNetworkError`

好处是：

1. 错误能标准化返回给客户端
2. 客户端可以按错误码做区分处理
3. 后续也方便埋点和统计

这比直接返回字符串或者直接抛异常更利于框架层治理。

---

## 六、性能优化与 benchmark

### 24. 你做过哪些性能优化？最有代表性的一个是什么？

参考回答：

我做过几类优化：

1. **连接复用**
   - 避免每次调用都重新建 TCP 连接

2. **`TCP_NODELAY`**
   - 减少小包场景下 Nagle 带来的额外延迟

3. **header/body 合并发送**
   - 避免一次 RPC 分两次小包发出

4. **线程池 + 有界队列**
   - 限制并发资源，避免过载拖死

其中最有代表性的是第 2 和第 3 个。

我一开始实现时，header 和 body 是分两次 `send` 的。结果 benchmark 里发现小 payload 的单次延迟明显偏高，而且重复调用场景特别明显。

后来我分析这很像典型的 **Nagle + delayed ACK** 问题，于是做了两件事：

- client/server 两端 socket 打开 `TCP_NODELAY`
- 把 header 和 body 合并成一次发送

改完之后，集成测试耗时从秒级下降到毫秒级，benchmark 也明显正常了。这是我项目里最有代表性的“测出问题 -> 定位 -> 优化 -> 验证”的闭环。

---

### 25. 你的 benchmark 是怎么设计的？为什么这样设计？

参考回答：

我设计 benchmark 时，想覆盖三个维度：

1. **基础吞吐和延迟**
   - `rpc_add_c1/c2/c4/c8`
   - 看并发度提升时 QPS 和延迟如何变化

2. **payload 大小影响**
   - `rpc_echo_64b_c4`
   - `rpc_echo_1024b_c4`
   - `rpc_echo_4096b_c4`
   - 看数据包变大后吞吐和尾延迟变化

3. **过载保护效果**
   - `rpc_delay_overload_c16`
   - 人为制造慢请求和队列压力，看系统是拖垮还是快速拒绝

同时我还加了一个：

- `local_add_baseline`

这个是进程内直接函数调用，主要用于对比“业务逻辑耗时”和“RPC 链路开销”。

这样设计的原因是：不仅要证明框架能跑，还要证明我知道怎么评价它。

---

### 26. 你 benchmark 里最有价值的数据是什么？

参考回答：

我觉得最有价值的数据有三类：

1. **小请求高并发能力**
   - `rpc_add_c8` 达到 `19892.11 QPS`
   - `p99` 约 `951.86 us`
   - 说明基础链路已经比较顺畅

2. **大 payload 下的性能变化**
   - `Echo` 从 64B 到 4KB，QPS 从 `18180.91` 降到 `14094.19`
   - 这很符合预期，说明瓶颈逐渐转向内存拷贝和内核收发成本

3. **过载保护数据**
   - 160 个请求里 60 个直接 `BUSY`
   - 不是无限排队
   - 这能说明框架不是只考虑 happy path，也考虑了稳定性

面试里我一般不会背所有数字，而是记住几个代表性指标和趋势。

---

### 27. 为什么 `rpc_add_c2` 的平均延迟比 `rpc_add_c1` 还低？

参考回答：

这个现象是可能出现的，原因主要有几个：

1. **连接预热和调度影响**
   - benchmark 是短时间测试，少量场景中系统调度抖动会比较明显

2. **吞吐提升后平均等待时间可能下降**
   - 在低并发时，如果单连接受到 delayed ACK 或系统调用节奏影响，平均值不一定最优

3. **样本量和环境核数有限**
   - 当前环境只有 2 个逻辑核
   - 所以一些数据会体现出“并发提升后 pipeline 更饱满”的特点

我在面试里会强调：

- 不只看单个点
- 要看整体趋势

整体趋势上，随着并发增加，QPS 上升、平均延迟和尾延迟最终也会逐渐增加，这符合系统行为预期。

---

### 28. 你怎么看待 benchmark 的局限性？

参考回答：

我认为这个 benchmark 是“对当前项目足够有价值”，但它也有明确局限：

- 压测机和服务端在同一台机器，本质是 loopback
- 只有 2 个逻辑核，不代表多核高性能环境
- 没引入真实业务逻辑和数据库
- 当前是同步调用模型，不代表异步框架上限

所以这些数据更适合回答两个问题：

1. 我的框架有没有明显设计错误
2. 做的优化有没有产生可观测收益

它不适合拿来和生产级 RPC 中间件做绝对性能 PK。

## 七、测试与验证

### 29. 你怎么验证这个框架是对的？

参考回答：

我做了三层验证：

1. **单元测试**
   - `tests/test_runtime.cpp`
   - 验证协议头编码解码、请求/响应体 round-trip、配置解析、静态注册发现

2. **集成测试**
   - `tests/test_rpc_integration.cpp`
   - 启动真实服务端，跑真实客户端调用
   - 验证正常调用、重复调用、鉴权失败、超时场景

3. **benchmark**
   - `benchmarks/benchmark_rpc.cpp`
   - 既测性能，也能帮助暴露并发和网络层问题

我很看重 benchmark，因为这个项目里最关键的 TCP 小包延迟问题，就是 benchmark 先暴露出来的。

### 30. 你为什么说 benchmark 也是一种验证手段？

参考回答：

因为很多框架问题单测未必能暴露出来，尤其是：

- 并发调度问题
- 网络时延问题
- 长连接复用问题
- 小包发送策略问题

我的例子就很典型：

- 功能测试一开始其实都能过
- 但 benchmark 跑起来后，发现小请求链路耗时明显不合理

最后我通过 benchmark 反向定位到了小包发送的实现问题。所以对 RPC 这种偏基础设施的项目来说，benchmark 不只是“跑分”，它还是一种诊断工具。

## 八、设计取舍与工程思考

### 31. 为什么服务端当前是“每连接一个线程”，而不是 epoll？

参考回答：

因为我这个项目的目标是“在合理复杂度内实现完整 RPC 闭环”。

如果一开始就上 epoll/reactor：

- 代码复杂度会明显上升
- 状态机、缓冲区管理、连接生命周期都更复杂
- 容易把项目重点从 RPC 机制本身转移到 IO 多路复用实现上

所以我当前选择的是更容易讲清楚的模型：

- accept 线程
- 每连接一个读写线程
- 业务线程池

这个模型足够支撑项目验证，也给后续演进留下了空间。面试里我会明确说，下一阶段最自然的升级方向就是 epoll 化。

---

### 32. 如果要把这个项目升级成更像生产版，你优先改什么？

参考回答：

我会按下面优先级改：

1. **接入真实 Protobuf**
   - 让消息定义和兼容性更规范

2. **抽象注册中心并接入 etcd**
   - 让服务发现从静态配置升级为动态发现

3. **改为 epoll/reactor**
   - 降低大量连接下的线程成本

4. **支持异步调用**
   - 返回 future 或 callback

5. **增加治理能力**
   - 重试、熔断、限流、指标、 tracing

我会先从 Protobuf 和注册中心开始，因为这两点能最快让项目从“原理实现”走向“工程化实现”。

---

### 33. 这个项目最大的技术难点是什么？

参考回答：

如果从编码复杂度看，不是代码生成，而是**网络行为和性能问题的定位**。

因为代码生成本质上是确定性逻辑，只要语法和模板设计清楚，实现难度可控。

真正更有挑战的是：

- TCP 字节流边界处理
- 长连接在异常场景下的状态一致性
- 小包请求为什么慢
- 超时后为什么要 reset
- 队列满时如何做快速拒绝

这些问题比“把代码写出来”更接近真实中间件开发的难点。

我最满意的是它形成了一个完整闭环：

## 九、追问型问题

### 35. 如果服务端处理逻辑执行很久，当前框架会发生什么？

参考回答：

当前模型下，请求会在线程池里执行。

如果服务端逻辑很慢：

- 线程池工作线程会被长期占用
- 后续请求会在队列里等待
- 如果队列满了，新请求会收到 `BUSY`

这说明当前框架已经有基础过载保护，但还没有做到更高级的能力，比如：

- 服务端执行中断
- 任务取消
- 优先级调度
- 自适应限流

这也是后续可以继续演进的方向。

---

### 36. 你现在的请求 ID 有什么作用？

参考回答：

当前主要有两个作用：

1. **请求和响应匹配**
   - 客户端发出请求时带 `request_id`
   - 收到响应时校验它和当前请求是否一致

2. **为后续异步化做准备**
   - 如果未来改成一个连接并发复用多个请求
   - `request_id` 就是做 response demultiplex 的关键字段

在当前同步实现里，它的价值更多是“协议完整性”和“未来扩展性”。

### 37. 为什么你说这个项目适合作为简历项目？

参考回答：

因为它同时具备了三个特征：

1. **主题明确**
   - 一眼能看出是 RPC 框架，不是零散 demo

2. **技术纵深足够**
   - 协议、代码生成、网络通信、并发、错误处理、性能优化都覆盖了

相比一些“只做业务页面”或者“只调用现成 SDK”的项目，这种基础设施类项目更容易体现底层能力和系统设计能力。
