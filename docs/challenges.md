# 实现中的重难点

## 1. 代码生成边界的设计

难点：

- 既要体现“IDL -> Stub -> 服务注册”的完整链路
- 又不能把项目复杂度抬到接近真正的 protobuf 编译器

解决方式：

- 设计了 protobuf 风格的精简 IDL 子集
- 生成最核心的四类代码：消息结构体、序列化函数、客户端 Stub、服务端胶水层

面试时可以强调：

- 这一步是为了把“业务接口定义”和“网络通信实现”分离
- 让业务开发只写 service 实现，不直接接触协议拼包逻辑

## 2. 自定义协议与边界切包

难点：

- TCP 是字节流，没有消息边界
- 必须保证服务端能准确区分一帧一帧的请求

解决方式：

- 设计固定长度帧头
- 帧头中显式携带 `body_length`
- 先读满 header，再按长度读取 body

## 3. 小包请求性能抖动

难点：

- 初版实现中，header 和 body 分两次 `send`
- Tiny payload 场景出现明显额外延迟

定位与优化：

- 原因是 `Nagle + delayed ACK` 触发了典型的小包延迟问题
- 优化 1：客户端和服务端 socket 打开 `TCP_NODELAY`
- 优化 2：把 header/body 合并为一次发送

这是项目里非常适合讲的性能优化故事，因为它和真实工程问题高度一致。

## 4. 并发与过载保护

难点：

- 如果每个请求都直接执行，服务端在高并发下容易出现排队失控
- 如果无限排队，调用方只会看到整体变慢

解决方式：

- 服务端使用固定大小线程池
- 线程池队列设置上限
- 队列满时立即返回 `BUSY`

价值：

- 比“无限堆积请求”更符合服务治理思路
- 在基准中可以直接展示过载保护效果

## 5. 超时与资源清理

难点：

- 客户端超时后，连接状态与服务端响应可能错位
- 如果不处理好，容易出现连接污染和错误复用

解决方式：

- 每次调用都设置 socket 读写超时
- 一旦读写失败或超时，客户端主动 `Reset`
- 服务端写关闭连接时屏蔽 `SIGPIPE`

## 6. 为什么没有直接接入 protobuf / etcd

原因：

- 这个项目的重点是展示 RPC 框架“核心能力”的理解和落地
- 如果首版直接引入过多外部依赖，会掩盖自己对协议、编解码、Stub、服务发现链路的掌控力

迭代方向：

- 下一步完全可以把 `StaticRegistry` 抽象替换成 etcd
- 把自研 IDL 子集替换成真实 protobuf + `protoc` 插件
