# MiniRPC 简历关键词高频面试问题整理

这份文档不是标准题库，而是专门围绕你简历里已经写出的关键词，去推测面试官最可能继续深挖的问题。核心目标是帮你提前覆盖两类问题：

- 面试官看到关键词后的第一反应追问
- 面试官看到性能数字、亮点描述后的真实性与深度追问

---

## 1. 开场高频题

这几题几乎一定会被问到，通常出现在“你先介绍一下项目”之后。

1. 你先整体介绍一下 `MiniRPC`，它解决了什么问题？
2. 为什么要自己实现一个 RPC 框架，而不是直接用 gRPC、brpc 或 thrift？
3. 这个项目的核心链路是怎样的？一次 RPC 调用从客户端到服务端经历了哪些步骤？
4. 你在这个项目里主要负责哪些模块？哪些部分是你自己从零设计的？
5. 你觉得这个项目最能体现你能力的点是什么？
6. 这个项目目前做到什么程度？哪些能力做了，哪些能力没做？

---

## 2. 按简历关键词拆解的高频追问

## 2.1 `C++11/14` / 从零实现

这组问题通常不是单独考语法，而是考你是否真的用语言特性完成了工程实现。

1. 你这个项目里主要用了哪些 `C++11/14` 特性？
2. 线程池、任务队列、连接处理这些模块里用了哪些并发原语？
3. 为什么这里选择 `std::thread` / `std::mutex` / `std::condition_variable`？
4. 你在哪些地方用了移动语义、右值引用或 `emplace` 优化？
5. 这个项目如果限制在 `C++11`，哪些实现会受影响？如果放宽到 `C++17/20`，你会怎么重构？
6. RAII 在这个项目里是怎么体现的？socket、线程、锁、缓冲区资源如何管理？
7. 你有没有遇到对象生命周期、悬空引用、并发竞态这类典型 C++ 问题？

## 2.2 `轻量级自研 RPC 框架`

这是项目标题里的核心词，面试官通常会先问“你理解的 RPC 到底是什么”。

1. 什么是 RPC？它和本地函数调用、本地 IPC、HTTP REST 的本质区别是什么？
2. 你这个 RPC 框架的最小闭环包含哪些模块？
3. 一个 RPC 框架为什么通常需要协议、序列化、服务发现、Stub、线程模型这些组件？
4. 你这个项目为什么叫“轻量级”？轻量体现在哪？
5. 如果让你定义这个项目的边界，你会怎么说？
6. 你觉得它和生产级 RPC 框架最大的差距是什么？

## 2.3 `自定义二进制协议`

这是最容易被连续深挖的关键词，通常会从协议格式一直问到粘包拆包、字节序、扩展性。

1. 你为什么要自定义二进制协议，而不是直接用 HTTP/JSON？
2. 你的协议头里有哪些字段？为什么要有 `request_id`、消息类型、负载长度这些字段？
3. 为什么采用“固定头 + 可变长包体”的设计？
4. 固定头长度是怎么定的？魔数、版本号、长度字段各自解决什么问题？
5. TCP 是字节流协议，你是怎么解决粘包拆包问题的？
6. 你协议里的长度字段为什么重要？如果长度字段异常怎么处理？
7. 你的协议有没有做大小端统一？为什么要这么做？
8. 请求和响应的编码格式分别是什么？错误响应如何表达？
9. 如果以后要加压缩、校验和、trace_id、扩展字段，你打算怎么扩展协议？
10. 自定义二进制协议相比文本协议的优缺点是什么？

## 2.4 `request_id` / 消息类型 / 负载长度 / 错误码体系

这些词一旦写进简历，面试官很容易逐个点名。

1. `request_id` 的作用是什么？同步调用里为什么还需要它？
2. 如果以后做连接复用下的并发请求，`request_id` 会怎么发挥作用？
3. 消息类型一般会区分哪些种类？请求、响应、错误还有什么？
4. 为什么协议层要有错误码，而不是只靠业务返回值？
5. 你的错误码如何分层？网络错误、框架错误、业务错误怎么区分？
6. `UNKNOWN_SERVICE`、`UNKNOWN_METHOD`、`AUTH_FAILED`、`TIMEOUT`、`BUSY` 分别在哪些阶段产生？
7. 如果服务端宕机、客户端超时、请求包非法，这三类错误会如何表现？

## 2.5 `精简 IDL`

这组问题核心是“为什么需要契约”和“为什么自己只做精简版”。

1. 为什么 RPC 框架需要 IDL？
2. 你的 IDL 支持哪些语法元素？
3. 为什么只实现精简子集，而不是完整兼容 protobuf？
4. 你这个 IDL 的 message、service、rpc 是怎么映射到生成代码里的？
5. 如果接口要演进，比如新增字段、字段缺失、版本兼容，你现在的 IDL 能不能支持？
6. 你觉得 IDL 在多人协作、前后端协作或跨服务协作中的价值是什么？

## 2.6 `mrpcgen` / 自动生成消息结构体、序列化函数、客户端 Stub、服务端胶水代码

这是很强的亮点，也很容易被怀疑“是不是只是模板拼接”。面试官一般会追问实现细节。

1. `mrpcgen` 的输入、输出分别是什么？
2. 你是怎么解析 IDL 的？词法分析、语法分析是手写的还是正则驱动的？
3. 为什么要做代码生成，而不是手写请求结构和 Stub？
4. 自动生成的代码具体包括哪些文件和内容？
5. 客户端 Stub 做了哪些事？和本地函数调用相比多了哪些隐藏逻辑？
6. 服务端胶水代码做了哪些事？如何把 `service + method` 映射到实际处理函数？
7. 生成器如何处理序列化和反序列化逻辑？
8. 你这个生成器最大的限制是什么？如果支持 `repeated`、嵌套消息、可选字段，该怎么扩展？
9. 为什么选择 Python 写生成器，而不是继续用 C++？
10. 如果让你对接真正的 `.proto` 和 `protoc`，你会怎么演进当前方案？

## 2.7 `同步调用链路`

同步调用看起来简单，但面试官经常借这个点问阻塞、超时、并发模型和未来演进。

1. 你为什么先做同步调用，而不是异步调用？
2. 同步调用链路中，客户端线程阻塞在哪一步？
3. 同步模型最大的优点和缺点是什么？
4. 如果一个连接上要支持多路并发请求，当前同步实现还能不能工作？
5. 你后续如果要扩展成异步 RPC，会改哪些模块？
6. 超时是在客户端控制还是服务端控制？为什么？

## 2.8 `静态服务发现`

面试官常用这组问题考你对“可用的最小实现”和“生产化差距”的理解。

1. 什么叫静态服务发现？
2. 你为什么先做静态服务发现，而不是 etcd、ZooKeeper、Consul？
3. 客户端是怎么根据服务名找到地址的？
4. 如果服务实例变更、扩缩容或故障迁移，静态服务发现有什么问题？
5. 你当前有没有抽象注册中心接口？后续替换动态注册中心难不难？
6. 静态服务发现更适合什么场景？

## 2.9 `鉴权`

鉴权是简历里的安全关键词，面试官会重点看你是不是只做了“字符串比较”。

1. 你这里的鉴权机制具体是什么？
2. token 放在协议的哪个位置？为什么放那里？
3. 鉴权是在服务端哪个阶段执行的？在反序列化前还是后？
4. 你的鉴权失败是如何返回的？
5. 这个鉴权方案有什么局限性？
6. 如果上升到生产级，你会如何升级鉴权能力？
7. 除了 token，你简历里如果提到 IP 白名单，具体是怎么校验的？

## 2.10 `超时控制`

这是很高频的工程追问，通常会问“超时到底由谁感知、怎么实现、和取消有什么区别”。

1. 你的超时控制是怎么做的？
2. 是 socket 超时、业务 deadline，还是两者都做了？
3. 客户端超时和服务端执行超时分别怎么处理？
4. 如果客户端超时了，但服务端业务还在执行，会发生什么？
5. deadline 和 timeout 有什么区别？
6. 为什么超时能力必须由框架层统一提供，而不是让业务自己控制？

## 2.11 `连接处理 + 业务线程池`

这组问题是并发模型核心，面试官会重点考“为什么这么拆”和“这样拆的代价是什么”。

1. 你为什么要把网络收发和业务执行解耦？
2. 你的服务端线程模型具体是怎样的？
3. 为什么不是单线程串行、每请求一线程，或者纯 reactor/epoll 模型？
4. 一个连接一个处理线程的优点和缺点是什么？
5. 业务线程池大小如何确定？
6. 任务从连接线程投递到业务线程池时，如何保证线程安全？
7. 如果业务执行很慢，会不会把连接线程拖死？
8. 这个模型在连接数非常大时有什么瓶颈？
9. 如果让你继续优化，你会不会改成 `epoll + 线程池`？

## 2.12 `固定大小线程池 + 有界队列 + 过载保护`

这是简历里非常容易被深挖的亮点，因为它和“稳定性设计”直接相关。

1. 为什么线程池要固定大小，而不是动态扩容？
2. 为什么任务队列要有界，而不是无界？
3. 有界队列满了以后为什么选择快速拒绝，而不是阻塞等待？
4. `BUSY` 这种快速失败策略的价值是什么？
5. 你怎么证明过载保护是生效的？
6. 如果没有有界队列，高压下系统会发生什么？
7. 过载保护和限流的区别是什么？
8. 如果要进一步做熔断、令牌桶限流、优先级队列，你会怎么设计？

## 2.13 `TCP 连接复用`

这类问题通常围绕性能和资源成本。

1. 为什么要做连接复用？
2. 如果每次调用都重新建连，会带来哪些问题？
3. 连接复用在你当前同步模型下是怎么工作的？
4. 一个连接是否允许同时承载多个未完成请求？
5. 连接断开、半开、异常关闭时你是怎么处理的？
6. 连接复用会不会带来并发安全问题？

## 2.14 `TCP_NODELAY`

这基本是性能优化故事的必问点。

1. 为什么要打开 `TCP_NODELAY`？
2. Nagle 算法是什么？它为什么会影响小包延迟？
3. `TCP_NODELAY` 带来的收益和代价分别是什么？
4. 你的 benchmark 能体现这个优化的效果吗？
5. 如果关闭 `TCP_NODELAY`，你预计哪些场景受影响最大？

## 2.15 `header/body 合并发送`

这个点说明你不是只会喊优化口号，而是实际做过微观优化。

1. 为什么把 header 和 body 合并发送？
2. 这个优化减少的到底是什么成本？系统调用次数、上下文切换还是网络包数量？
3. 它和 `TCP_NODELAY` 的关系是什么？
4. 如果数据量很大，这种合并发送方式还有没有价值？
5. 有没有考虑过 `writev` / `sendmsg` 这类 scatter-gather 方案？

## 2.16 `19892 QPS` / `p99 < 1 ms` / `并发 8`

简历里只要写了具体性能数字，面试官几乎一定会追问真实性、环境、指标含义和边界条件。

1. `19892 QPS` 是在什么环境下测出来的？机器配置、核数、编译选项是什么？
2. 为什么选 `Add` 场景做基准？
3. `p99 < 1 ms` 具体测的是哪一段延迟？端到端、网络、还是业务函数本身？
4. 为什么并发 8 时 QPS 最好？再往上为什么没继续明显提升？
5. 这些数字和 payload 大小的关系是什么？
6. 为什么 `Echo 4KB` 场景 QPS 会下降？
7. 你怎么保证 benchmark 不是偶然结果？有没有重复测试或固定环境？
8. 本地函数调用和 RPC 调用的性能差距有多大？瓶颈主要在哪？
9. 你做过哪些优化后，指标有明显改善？

## 2.17 `37.5% 请求被快速拒绝而非阻塞超时`

这是很好的稳定性故事，面试官常常会围绕它问设计动机与系统行为。

1. 这个 `37.5%` 是怎么测出来的？
2. 为什么你认为“快速拒绝”比“排队等待直到超时”更合理？
3. 快速拒绝对调用方有什么好处？
4. 如果请求被拒绝，客户端应该如何处理？重试还是失败返回？
5. 成功请求为什么还能保持接近预期服务时间？
6. 这个结果说明了线程池和有界队列的什么特性？
7. 如果把队列加大，是不是拒绝率会下降？为什么这不一定是好事？

## 2.18 `单元测试 / 集成测试 / benchmark / 性能报告`

这组问题主要看你是否形成了工程闭环。

1. 你这个项目的测试分层是怎样的？
2. 单元测试覆盖了哪些核心模块？
3. 集成测试主要验证哪些场景？
4. 鉴权失败、超时、重复调用、服务发现这些场景有没有测？
5. benchmark 和普通功能测试有什么区别？
6. 为什么性能报告也值得写进简历？
7. 你在测试里发现过什么 bug？是怎么修的？
8. 你如何保证测试结果可复现？

## 3. 面试官最可能继续追打的“为什么”

如果面试官觉得你项目写得不错，通常会从“你为什么这么设计”继续往下追。

1. 为什么协议头要固定长度，而不是全部 TLV？
2. 为什么先做同步 RPC，而不是一步到位异步化？
3. 为什么先做静态服务发现，而不是动态注册中心？
4. 为什么生成器用 Python，而运行时用 C++？
5. 为什么选择二进制协议，而不是 JSON？
6. 为什么服务端采用“连接处理 + 业务线程池”，而不是 reactor？
7. 为什么高压下要快速拒绝，而不是尽量排队保住成功率？
8. 为什么你的项目强调“完整闭环”，而不是只做一个 socket demo？

3. 你这个协议和序列化方式在兼容性上有哪些风险？
4. 你的线程模型在高连接数下会不会撑不住？
5. 你的服务发现、鉴权、超时、错误处理里，哪个最脆弱？
6. 如果让你和 gRPC、brpc 对比，你的优势和劣势分别是什么？
7. 这个项目里最难的 bug 是什么？你怎么定位的？
8. 如果再给你一周，你会优先优化哪里？

## 6. 建议你的回答方式

准备这些问题时，尽量按同一个结构回答，面试里会更稳：

1. 先说结论
2. 再说为什么这么设计
3. 再说具体实现
4. 最后补一句收益、限制或后续演进

例如回答性能题时，不要只背 `19892 QPS`，而要顺带说明：

- 测试环境是什么
- 测的是哪个 benchmark
- 延迟指标覆盖哪一段链路
- 做过哪些优化
- 这个结果说明了什么

例如回答框架设计题时，不要只说“这么做更好”，而要补上：

- 替代方案是什么
- 为什么当前阶段没选那个方案
- 当前方案的边界和缺点是什么

---

## 7. 一句话结论

基于这份简历，面试官最可能围绕四条主线来提问：

- 你是否真的理解 RPC 的完整调用链路
- 你是否真的做过协议、代码生成、并发模型这些核心实现
- 你写在简历里的性能数字是否真实、可解释、可复现
- 你是否清楚自己方案的取舍、边界和后续演进方向
